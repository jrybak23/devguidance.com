RDMSs Isolation Level Comparison

April 7, 2022

Isolation level analyser#

This tool was developed for getting results for this article. Which generates such tables per each implementation of this interface.

import org.hibernate.dialect.Dialect;
import org.hibernate.dialect.PostgreSQL95Dialect;
import org.springframework.stereotype.Component;
import org.testcontainers.containers.JdbcDatabaseContainer;
import org.testcontainers.containers.PostgreSQLContainer;

@Component
public class PostgreSQL14 implements DatabaseToAnalyze {
    @Override
    public JdbcDatabaseContainer<?> createContainer() {
        return new PostgreSQLContainer<>("postgres:14");
    }

    @Override
    public Class<? extends Dialect> getDialect() {
        return PostgreSQL95Dialect.class;
    }
}

You can view all results here https://jrybak23.github.io/isolation-levels-analyzer/.

Results may not be 100% correct!#

So please check these analysis implementations before making some conclusions.

Conclusions#

  • The behavior of different RDMSs for transaction isolation may differ significantly.
  • Ways of preventing some phenomenas be reproducible are also different. Some RDMSs rollbacks transaction, some block transaction execution until another transaction release the lock…
  • Set lock timeout if you use MS SQL Server with JDBC. Or jakarta.persistence.query.timeout hint in case of JPA. Otherwise, transaction can be infinitely locked.
