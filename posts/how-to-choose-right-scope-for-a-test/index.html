How to choose right scope for a test

September 5, 2021

Test scope introduction#

Rather talk about specific definition of test scope let’s review list of different test types ordered from higher to lower scope.

Basically, the more things like methods, endpoints, databases, services, apps are involved in test the higher is scope.

How to choose right scope for test#

There is no clear recommendation how to choose scope for test. Instead of reeling on approximate count concept like test pyramid which inherently makes sense, but besides there are definitely more things that should be considered while choosing test type. I would rather recommend try to trade off between reasons described below. Some of them are really obvious, but they are mentioned just to provide more or less full picture what should be taken into account for choosing scope.

Reasons to choose higher scope#

Basically, higher scope test gives more confidence that app works as expected. Here are more detail described reasons to choose higher scope tests:

1. Test required behaviour rather than implementation details#

It desired that test should be resilient to refactor. There is a bunch of ways how to implement certain logic. By adding test to protect the logic, we expect that it fails only when required behavior is broken. E.g. GET endpoint that returns some user data loaded from database.

There are a bunch of:

  • Frameworks for HTTP request handling.
  • Ways how to implement/split services, repositories and other classes.
  • Technologies that can be used for persistence. E.g. ORM frameworks, databases.

It’s expected that all this thing described above can be changed over time. By choosing lower scope it may increase coupling between test and implementation. So it becomes harder to evolve system over time.

2. Tests that intended to cover business logic only doesn’t cover compatibility business logic with lower layer#

E.g. by mocking SQL queries, ORM calls, or other parts of persistence layer, it doesn’t protect the correctness of calling the persistence layer.

3. No reason to be concerned about performance of tests beforehand in case when backing services are involved to test#

In most cases, by running some backing services in Docker (e.g. using testcontainers) doesn’t take reasonable amount of time to set up environment for tests or doesn’t significantly slow down the test. So taking into account the reason #2 you likely benefit from it.

Reasons to choose lower scope#

1. Issues with state in high scope tests like API o UI tests#

It may be complicated to set up initial consistent state for them before each test/suite. Such set up may slow down testing. At same time, some services may be down or overloaded. So that’s why count of such tests expected to be significantly lower than integration+unit tests count.

2. Generic API#

Some parts of the app expected to reuse in many other parts of the app. They are intended to help to implement business logic rather to implement business logic itself.

Let’s take a look on example of the class that provides generic API.

import java.time.LocalDate;
import java.util.function.Predicate;

public class Interval {

    private final LocalDate begin;
    private final LocalDate end;

    public static Interval between(LocalDate begin, LocalDate end) {
        return new Interval(begin, end);
    }

    /**
     * @param predicate function that accepts date and returns true - if the date should be counted
     * @return count of days in the interval filtered by predicate function.
     */
    public long countDays(Predicate<LocalDate> predicate) {
        long result = 0;
        for (LocalDate date = begin; !date.isAfter(end); date = date.plusDays(1)) {
            if (predicate.test(date)) {
                result++;
            }
        }
        return result;
    }

    private Interval(LocalDate begin, LocalDate end) {
        this.begin = begin;
        this.end = end;
    }
}

It can be reused differently to implement logic in many places. Like this.

    long workdays=Interval.between(startDate,endDate)
        .countDays(this::isNotHolidayOrWeekend);

But it’s very unlikely that business logic will change that way, so it’ll require changing logic of counting days. So we can confidently cover it by unit test without warring of increasing coupling.

3. Cases when needed to add mock or stub#

There is no direct correlation between how high is scope and count of mocks. It should rather happen because of:

  1. Achieving consistency by avoiding external calls. HTTP requests to other service or consuming/publishing some data to from/into a real queue should be mocked, stubbed or can be replaced by testcontainers.

  2. When increasing scope requires additional excessive test setup. Which leads to common test anti-patterns like: Excessive Setup, Mockery, Giant. For example, additional setup of module or mocks. From my experience, it may happen because of bad design of library/framework/platform.

  3. Avoid logic that takes reasonable amount of time to execute. That rather related to UI testing e.g. running animations. It’s not case when you save milliseconds per test if mock repository call instead of calling DB that runs in testcontainers.

  4. As the way of assertion by intercepting the call of method. When the result from method call can’t be asserted but needed to make sure that method was called.

  5. Logic that depends on current time.

Conclusion#

The article likely doesn’t cover all things about test scope. But main point is instead of sticking to one of concepts try to be flexible and take into account much more things while choosing approach for testing.
